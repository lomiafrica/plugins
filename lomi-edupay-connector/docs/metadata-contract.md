# EduPay metadata contract

All charges MUST include these fields so EduPay can reconcile payments.

## Required on every fee payment

| Field | Type | Example | Notes |
| --- | --- | --- | --- |
| `payment_reference` | string | `FEE-2026-TERM1-00042` | Unique per fee attempt; drives idempotency |
| `school_id` | string | `school_lycee_abidjan` | EduPay school primary key |
| `student_id` | string | `STU-8842` | EduPay student primary key |
| `fee_code` | string | `TUITION_TERM1` | Fee type / product code |
| `term_id` | string | `2026-T1` | Optional but recommended |

## Customer (payer)

| Field | Required | Notes |
| --- | --- | --- |
| `customer.name` | Yes | Parent / guardian name |
| `customer.phone` | Yes | E.164 — required for MoMo |
| `customer.email` | Card only | Required for `POST /charge/card` |

## Storage per rail

| Rail | `payment_reference` | Other metadata |
| --- | --- | --- |
| Card | `payment_reference` field + `metadata` object | Full metadata on charge |
| Wave / MTN | Embedded in `description` | Card-style metadata on MoMo API is planned; use description until then |

Description format (generated by `@edupay/lomi-client`):

```
EduPay fee {payment_reference} school={school_id} student={student_id} fee={fee_code} term={term_id}
```

## Idempotency

Header: `Idempotency-Key: edupay_{payment_reference}`

Safe to retry the same fee creation; do not reuse references across different fees.

## Webhook matching

Prefer, in order:

1. `payload.data.metadata.payment_reference` (card)
2. `payload.data.metadata.school_id` + amount + timestamp
3. Parse `description` for Wave/MTN until metadata is extended on those rails
